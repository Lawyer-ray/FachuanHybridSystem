## 现象与根因归纳（用于指导后续整改顺序）

### 1) 严格模式下的存量缺注解（no-untyped-def）
- 现象：services/config 等目录存在大量未标注参数/返回类型的方法与函数。
- 影响：一旦补齐注解，会连锁暴露更多 `arg-type/return-value`，需要按“先边界后内部”的顺序修。
- 建议：优先补齐门禁覆盖文件的关键边界（Service/Adapter/Repo 的 public 方法），并尽量使用 DTO/Protocol 收紧 `Any` 传递链路。

### 2) 动态属性/运行期注入导致 attr-defined
- 现象：mixin/运行期注入的属性（例如 `self.page`、`navigate_to_url()` 等）在类型层面没有来源。
- 影响：错误量大且分散，容易让修复陷入“到处 ignore”。
- 建议：用 Protocol 或基类显式声明属性/方法契约，把动态注入收敛到少量入口处。

### 3) Any 外溢导致 no-any-return（门禁口径的主因）
- 现象：第三方库缺少 stub、`--follow-imports=skip`、以及 ORM/ServiceLocator 的动态返回，都会让表达式退化为 `Any`。
- 影响：严格返回类型（例如 `-> str` / `-> HttpResponse` / `-> list[...]`）下频繁触发 `no-any-return`。
- 建议：优先在“返回点”做类型收敛（`cast(...)`/显式构造对象/显式转换），并逐步消灭上游 `Any` 来源（例如补齐 adapter/service 的返回类型）。

### 4) Protocol/Interface 双轨并存导致类型不兼容（arg-type/override）
- 现象：全量输出中出现 `apps.core.protocols.*` 与 `apps.core.interfaces.*` 的同名接口互不兼容（即使方法签名看起来一致，也会被视为不同类型）。
- 影响：依赖注入/构造参数会出现“传了一个接口，却不是期望的那个接口”的错误，阻断大范围修复。
- 建议：统一对外契约出口（选择 protocols 或 interfaces 作为唯一来源），并提供兼容 alias 或逐步迁移方案，避免两套类型并行扩散。

### 5) django-stubs mypy plugin（当前阻塞项）
- 结论：当前仓库依赖锁定 `mypy==1.8.0` + `django-stubs==5.2.9` 的组合下，启用 `mypy_django_plugin` 会触发 mypy INTERNAL ERROR（指向 `django-stubs/http/request.pyi:56`），导致 typecheck 直接中断。
- 额外问题：启用 plugin 会真实初始化 Django（包括 admin autodiscover），因此项目内任何“import side-effect / ImportError”都会放大为 mypy 运行失败；本轮已修复 `apps.automation.services` 的导出别名，避免 import 直接崩溃，但 INTERNAL ERROR 仍需通过升级 mypy 解决。
- 建议：先以“门禁口径修复（不启用 plugin）”为主线清零错误；随后再评估升级 mypy（并同步更新 constraints / pre-commit / CI lock）来重新启用 plugin，以换取 ORM 类型精度提升。

## 修复模式

### 1. 移除错误的 -> None
```python
# 修复前
def get_service() -> None:
    return ServiceLocator.get_service()

# 修复后
def get_service() -> Any:
    return ServiceLocator.get_service()
```

### 2. 添加参数类型
```python
# 修复前
def process(self, data, user):
    pass

# 修复后
def process(self, data: Any, user: Any):
    pass
```

### 3. 添加 Optional 返回类型
```python
# 修复前
def get_item(self, id: int):
    if not found:
        return None
    return item

# 修复后
def get_item(self, id: int) -> Optional[Any]:
    if not found:
        return None
    return item
```

### 4. 修复 @property 返回类型
```python
# 修复前
@property
def service(self):
    return self._service

# 修复后
@property
def service(self) -> Any:
    return self._service
```

## 技术细节

### 使用的类型注解
- `Any`: 用于无法精确推断的类型
- `Optional[T]`: 用于可能返回 None 的函数
- `dict[str, Any]`: 用于字典参数
- `list[T]`: 用于列表参数

### 导入管理
所有修复自动添加必要的 typing 导入：
```python
from typing import Any, Optional
```

### 兼容性
- 使用 `from __future__ import annotations` 支持新式类型语法
- 兼容 Python 3.10+ 的类型注解
