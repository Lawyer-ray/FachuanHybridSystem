# 重构最佳实践

本文档用于约束后端重构的落地方式，核心原则是：**不破坏边界、不引入隐式耦合、可回归、可回滚**。

## 0. 重构前的判断

- 先确认目标是“降复杂度”还是“扩能力”
  - 降复杂度：优先拆职责、收敛依赖、补测试护栏
  - 扩能力：优先把扩展点抽到 Protocol/策略层，再落实现有能力
- 优先选择“兼容性重构”
  - 新旧入口并存（Facade 方式），内部转发到新实现
  - 用结构测试/单测保证不回退

## 1. 分层与职责边界（建议）

- Coordinator/Orchestrator：只负责流程编排，不写 ORM
- Service：聚合根入口（权限校验/关键业务校验/核心 CRUD）
- Workflow：一个事务里跨多表/跨聚合的复合流程
- Repo：集中 ORM 细碎逻辑（去重、批量查询、状态判定）
- Adapter：对接 Protocol/DTO（胶水层，不写业务第二套）
- Utils：纯函数工具，不产生数据库副作用

## 2. 跨模块交互（必须收敛）

- 在 `apps/*/services/**` 中，避免跨模块直接 import models
  - 优先通过 Protocol/DTO + ServiceLocator 的方式跨域协作
  - 少数只读、低耦合场景可用 `apps.get_model(...)` 延迟取模型，但要保证不引入循环依赖

## 3. 拆巨石文件的标准做法

当某个 service 文件出现以下任一情况，建议立即拆分：

- 超过 ~400 行，且包含 2 个以上职责（查询 + 写入 + 编排混杂）
- 大量重复的 ORM 片段散落在多个方法里
- “改一个地方要同时理解多个子域”的强耦合

推荐拆分顺序（降低风险）：

1. 先抽 Utils/Repo（纯函数/ORM 细碎逻辑）
2. 再抽 Processor/Workflow（复合流程）
3. 最后抽 Coordinator（入口编排）并让旧 service 作为 Facade

## 4. 测试与护栏（重构必须绑定）

- 结构护栏：优先补 `backend/tests/structure/`（约束导入、目录、越权调用）
- 单元测试：覆盖“可变逻辑”和“边界条件”
- 回归策略：每迁移一段逻辑就让测试绿一次，避免最后一次性大爆炸

## 5. 日志与排障（重构后必须更可观测）

- 禁止 `print`，统一使用 logger
- 不记录敏感信息（token、cookie、密钥、账号明文等）
- 日志尽量结构化（`extra={...}`），并携带 request_id/trace_id
