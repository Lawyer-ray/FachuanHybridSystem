# 可观测性 SLO、告警与 Runbook 验收

本文件定义“10/10 可观测性”的最小验收集合：指标可用、告警有效、排障可落地。

## 1. 指标基线（Golden signals）

### 1.1 API（HTTP）

必须具备：

- 吞吐：RPS（按路由/方法/组织维度）
- 延迟：P50/P95/P99（按路由/方法）
- 错误：5xx/4xx 比例（按错误码、业务错误码）
- 饱和度：CPU/内存/线程/连接池/DB 连接

### 1.2 异步任务（队列/worker）

必须具备：

- 队列长度（等待/运行）
- 任务耗时分布（P50/P95/P99）
- 失败率、重试率、取消率
- 超时率、死信数（若有死信机制）

### 1.3 外部依赖（HTTPX/第三方）

必须具备：

- 按 host 的成功率、超时率、平均耗时与分位数
- 关键依赖的降级/断路触发次数

## 2. SLO（Service Level Objectives）

建议至少定义 3 条 SLO，并明确统计窗口（7/28 天）与告警策略：

| SLO | 指标定义 | 目标 | 窗口 | 备注 |
|---|---|---:|---|---|
| API 可用性 | `2xx/3xx` 占比 | 99.9% | 28d | 以用户请求为口径 |
| API 延迟 | P95 < 800ms | 99% | 28d | 按关键路由分开 |
| 任务成功率 | 成功 / (成功+失败) | 99% | 28d | 关键队列单列 |

验收标准：

- 每条 SLO 有对应“错误预算”解释与处置流程
- 发生 SLO 退化时可在 5 分钟内定位到“哪个路由/哪个依赖/哪个队列”

## 3. 告警（Alerting）

### 3.1 必须存在的告警（至少 5 条）

- API 5xx 错误率突增（按路由/组织维度聚合）
- API P95 延迟突增（按关键路由）
- worker 队列积压（队列长度或等待时长超过阈值）
- DB/Redis 连接异常或不可用
- 外部依赖超时率突增（按 host）

### 3.2 告警质量要求

- 每条告警必须链接到对应 Runbook
- 告警必须包含最关键的上下文：request_id 示例、错误码 TopN、受影响路由/组织
- 告警必须可抑制：部署窗口、已知事故窗口

## 4. Runbook（运行手册）

每条告警至少包含以下模板段落：

1. 现象与影响范围（如何判断严重性）
2. 立即止血（5 分钟内可执行的动作）
3. 定位步骤（日志/指标/trace 的查询路径）
4. 回滚与恢复（含迁移/配置变更回滚）
5. 根因修复与复盘输出（要补的测试/监控/门禁）

## 5. 追踪与关联（Tracing/Correlation）

验收标准：

- 所有请求日志必须带 request_id
- 外部请求（HTTPX）与后台任务日志可关联到同一 request_id 或 trace_id
- 关键业务错误必须带稳定的业务错误码（可用于聚合告警）
