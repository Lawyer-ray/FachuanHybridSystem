# 证据管理系统深度分析报告

> 三角色交叉讨论：诉讼律师 × 产品经理 × 程序员
> 生成日期：2026-02-28

---

## 一、讨论背景

**议题**：如何在法穿AI系统中构建完整的证据管理能力——从接收、识别、整理到提交法庭，以及开庭时快速检索。

**现状**：系统已有 `evidence_sorting` 模块，实现了对账单/出库单的 OCR 分类、LLM 解析、交叉比对和 ZIP 导出。但这只覆盖了"买卖合同纠纷"中的部分证据场景，距离完整的证据管理体系还有很大差距。

**三个角色**：
- 🔷 **律师（L）**：10年+诉讼经验，熟悉民商事诉讼全流程
- 🟢 **产品经理（PM）**：法律科技产品专家，关注用户体验和商业价值
- 🟠 **程序员（DEV）**：系统架构师，熟悉 Django + AI 技术栈

---

## 二、核心讨论记录（精华摘要）

### 第一阶段：证据接收（Round 1-25）

**L**：证据来源极其分散——当事人微信发图、邮件附件、纸质扫描件、法院电子送达、对方当事人提交的证据。最头疼的是当事人一次性甩过来几百张手机照片，没有任何说明，你得自己猜这是什么。

**PM**：所以第一个核心需求是"多渠道统一收件箱"。不管证据从哪来，都汇聚到一个地方。

**DEV**：现有系统已经有飞书群消息解析、法院短信解析。可以扩展为统一的"证据收件箱"模式。技术上需要：
1. 文件上传 API（支持批量、大文件分片）
2. 飞书/微信消息中的图片自动抓取
3. 邮件附件自动解析（IMAP 监听）
4. 法院电子送达文书自动归档

**L**：还有一个关键点——当事人发来的证据必须记录"原始状态"。法官会问"这个证据是什么时候取得的"，如果你说不清楚，证据的证明力会打折扣。

**PM**：所以每份证据入库时必须自动记录：接收时间、来源渠道、原始文件哈希值（防篡改）、接收人。这是证据链的起点。

**DEV**：这个好实现。文件上传时计算 SHA-256 哈希，存入数据库。后续任何修改都不影响原始记录。可以用 Django 的 `FileField` + 自定义 `Storage` 后端。

**L**：对方提交的证据也要管理。开庭前法院会组织证据交换，对方的证据清单和证据原件都需要录入系统，方便后续逐一质证。

**PM**：这引出了"证据方向"的概念——我方证据 vs 对方证据 vs 法院调取证据。三种来源，管理方式不同。

**DEV**：数据模型上加一个 `direction` 字段：`plaintiff`（我方举证）、`defendant`（对方举证）、`court`（法院调取）。

**L**：补充一点，同一个案件可能有多轮举证。一审举证期限内提交一批，对方质证后补充一批，二审又补充一批。每一轮都要清晰区分。

**PM**：所以需要"举证轮次"的概念。每轮举证是一个独立的集合，有自己的截止日期和提交状态。

**DEV**：模型设计：`EvidenceRound`（举证轮次）→ 包含多个 `EvidenceItem`（证据项）→ 每个证据项包含多个 `EvidenceFile`（证据文件）。三层结构。

### 第二阶段：证据识别与分类（Round 26-50）

**L**：证据分类是最核心的专业能力。民事诉讼中证据分为：书证、物证、视听资料、电子数据、证人证言、鉴定意见、勘验笔录、当事人陈述。但实务中更重要的是按"证明目的"分类——这个证据是证明什么事实的。

**PM**：现有的 `evidence_sorting` 模块用 OCR + 关键词做分类，只能区分对账单/出库单/收款凭证。要扩展到通用场景，需要更强的 AI 能力。

**DEV**：可以分两层：
1. **文件类型识别**（技术层）：合同、发票、银行流水、聊天记录截图、身份证、营业执照、判决书……用多模态 LLM 识别
2. **证明目的标注**（业务层）：律师手动标注或 AI 建议，如"证明双方存在买卖合同关系"、"证明被告已收货"

**L**：第二层才是关键。法官看证据清单，第一眼看的就是"证明目的"。写得好不好直接影响法官对案件的第一印象。我见过太多律师写"证明案件事实"这种废话。

**PM**：那我们可以做一个"证明目的智能建议"功能——根据案由（如买卖合同纠纷）和证据类型，AI 自动生成证明目的的建议文本，律师确认或修改。

**DEV**：这个可以用 LLM 实现。Prompt 设计：给定案由 + 证据类型 + OCR 文本摘要，生成 2-3 个证明目的建议。现有的 `litigation_ai` 模块已经有 LLM 调用基础设施。

**L**：证据还有一个重要属性——"证据三性"：真实性、合法性、关联性。我方提交的证据要提前准备好三性说明，对方的证据要准备质证意见（通常是攻击三性中的某一个）。

**PM**：所以每个证据项需要两个文本字段：`proof_purpose`（证明目的）和 `three_properties_note`（三性说明/质证意见）。

**DEV**：对于对方证据，还需要一个 `cross_examination` 字段存质证意见。可以做成结构化的：
```
{
  "authenticity": {"opinion": "认可/不认可", "reason": "..."},
  "legality": {"opinion": "认可/不认可", "reason": "..."},
  "relevance": {"opinion": "认可/不认可", "reason": "..."}
}
```

**L**：非常好。实务中质证就是围绕这三个维度展开的。如果系统能帮我提前把质证意见的框架搭好，开庭时我只需要微调就行。

**PM**：AI 辅助质证意见生成——这是一个高价值功能。根据证据内容和案件背景，自动生成质证意见草稿。

**DEV**：技术上可行。输入：对方证据的 OCR 文本 + 案件基本信息 + 我方主张。输出：结构化质证意见。用 `litigation_ai` 的 chain 模式实现。

**L**：证据编号也很重要。法院要求证据清单上每份证据有唯一编号，格式通常是"证据1"、"证据2"……但实务中经常需要调整顺序，编号就乱了。

**PM**：系统应该支持拖拽排序，编号自动重新生成。这是基本的用户体验。

**DEV**：前端用拖拽排序，后端存 `sort_order` 字段。编号在导出时动态生成，不存数据库。

### 第三阶段：证据整理与组织（Round 51-75）

**L**：证据整理的核心输出是"证据清单"——这是提交给法院的正式文件。格式通常是表格：序号、证据名称、证据类型、证明目的、页码范围、原件/复印件。

**PM**：系统需要自动生成这个表格，支持 Word/PDF 导出。现有的 `documents` 模块已经有 docx 模板生成能力，可以复用。

**DEV**：复用 `documents` 模块的 `docx_templates` 基础设施。新增一个"证据清单"模板，数据从 `EvidenceItem` 模型聚合。

**L**：证据整理还有一个大痛点——页码标注。法院要求证据装订成册，每份证据标注页码范围。比如"证据1：合同（第1-15页）"、"证据2：发票（第16-18页）"。手动数页码非常痛苦。

**PM**：如果每份证据文件都有页数信息，系统可以自动计算累计页码。PDF 文件可以直接读页数，图片每张算1页。

**DEV**：用 `PyPDF2` 或 `pikepdf` 读 PDF 页数。图片文件每个算1页。模型加 `page_count` 字段，证据清单导出时自动计算 `start_page` 和 `end_page`。

**L**：还有一个实务需求——证据分组。复杂案件可能有几十份证据，需要按争议焦点分组。比如"关于合同成立的证据"、"关于违约事实的证据"、"关于损失金额的证据"。

**PM**：引入"证据组"概念，一个证据可以属于多个组（多对多关系）。组可以对应案件的争议焦点。

**DEV**：`EvidenceGroup` 模型，与 `EvidenceItem` 是 M2M 关系。可以关联到案件的 `DisputeFocus`（争议焦点）模型——如果有的话。

**L**：证据副本也是个问题。提交法院需要正本一份、副本若干份（按对方当事人数量）。系统应该能告诉我"这个案件需要准备几套证据"。

**PM**：这个信息可以从案件的当事人数量自动计算。在证据清单页面显示"需准备 X 套（正本1套 + 副本 Y 套）"。

**DEV**：从 `cases` 模块的当事人关联中计算。简单的聚合查询。

**L**：证据原件管理也很关键。有些证据只有复印件，开庭时法官可能要求出示原件。系统需要标记每份证据是"有原件"还是"仅复印件"，以及原件存放位置。

**PM**：每个 `EvidenceItem` 加 `has_original` 布尔字段和 `original_location` 文本字段。在开庭准备清单中高亮"需要带原件"的证据。

**DEV**：简单字段扩展，没有技术难度。

### 第四阶段：提交法庭（Round 76-90）

**L**：现在很多法院支持电子提交——通过"人民法院在线服务"或各地的电子诉讼平台。但格式要求各不相同。有的要求单个 PDF，有的要求按证据编号分别上传。

**PM**：系统需要支持多种导出格式：
1. 全部合并为一个 PDF（带目录和页码）
2. 按证据编号分别导出 PDF
3. 证据清单 Word 文档
4. 全套打包 ZIP

**DEV**：合并 PDF 用 `pikepdf` 或 `reportlab`。图片转 PDF 用 `Pillow`。目录页用 `reportlab` 生成。现有的 `exporter.py` 已经有 ZIP 导出逻辑，扩展即可。

**L**：电子提交还有文件大小限制，通常单个文件不超过 50MB。如果证据太多，需要自动分卷。

**PM**：导出时自动检测文件大小，超过阈值自动分卷。每卷生成独立的目录页。

**DEV**：分卷逻辑：遍历证据列表，累计文件大小，达到阈值时切分。每卷独立编号。

**L**：提交后需要记录提交状态——已提交/未提交/法院已签收。有些法院会发送签收回执，系统应该能关联。

**PM**：`EvidenceRound` 加提交状态字段和提交时间。如果能对接法院系统的回执就更好了。

**DEV**：现有的 `automation` 模块已经有法院系统对接能力（短信解析、文书下载）。可以扩展为监听提交回执。

### 第五阶段：开庭快速检索（Round 91-108）

**L**：这是最关键的场景。开庭时法官问"你方证据3是什么"，你必须在3秒内找到并回答。如果翻半天找不到，法官会非常不耐烦，对你的专业形象也是打击。

**PM**：开庭模式需要一个专门的界面——极简、快速、大字体。核心功能：
1. 按编号快速跳转（输入"3"直接定位到证据3）
2. 全文搜索（搜"合同"找到所有包含合同的证据）
3. 按证明目的筛选
4. 按争议焦点筛选

**DEV**：前端做一个"开庭模式"页面。数据预加载到内存，搜索用前端 JS 实现（不走后端 API），保证毫秒级响应。

**L**：法官还经常问"你方有没有关于XX的证据"，这时候需要按内容搜索。比如法官问"有没有2023年5月的送货记录"，我需要快速找到对应的出库单。

**PM**：这就需要 OCR 文本的全文索引。每份证据的 OCR 文本都要可搜索。

**DEV**：两个方案：
1. 简单方案：SQLite FTS5 全文索引，够用且零依赖
2. 进阶方案：向量检索（语义搜索），用现有的 LLM 基础设施生成 embedding

考虑到开庭场景对速度的极端要求，推荐方案1。FTS5 在 SQLite 上查询速度是微秒级的。

**L**：开庭时还有一个场景——对方突然提交新证据（虽然程序上不合规，但实务中经常发生）。我需要快速拍照录入，然后对照我方证据进行质证。

**PM**：开庭模式加一个"快速录入"按钮——拍照 → OCR → 自动分类 → 临时编号。不需要完整的证据录入流程，先存下来，庭后再整理。

**DEV**：复用现有的 OCR + 分类服务。前端调用手机摄像头拍照，上传后异步 OCR，结果实时显示。

**L**：还有一个高级需求——证据关联图谱。复杂案件中，证据之间是有关联的。比如"合同"关联"发票"关联"银行流水"，形成一条完整的交易链。如果法官质疑某个环节，我需要快速展示整条链。

**PM**：这是一个差异化功能。用图谱可视化展示证据之间的关联关系。

**DEV**：数据模型上加 `EvidenceRelation`（证据关联）表，存储两个证据之间的关系类型（如"合同→发票"、"发票→付款凭证"）。前端用 D3.js 或类似库做图谱可视化。但这个优先级可以放后面。

**L**：最后一个关键点——庭审笔记。开庭时我需要快速记录法官的问题和我的回答，并关联到具体证据。庭后整理时这些笔记非常有价值。

**PM**：开庭模式加"快捷笔记"功能。每条笔记可以关联一个或多个证据项。支持语音输入（手机端）。

**DEV**：`HearingNote` 模型，关联 `Case` 和 `EvidenceItem`（M2M）。语音输入用浏览器的 Web Speech API，零后端成本。

## 三、核心结论与建议

### 3.1 数据模型设计

```
Case (案件)
  └── EvidenceRound (举证轮次)
        ├── round_number: 第几轮
        ├── deadline: 举证期限
        ├── status: 草稿/已提交/已签收
        ├── submitted_at: 提交时间
        └── EvidenceItem (证据项)
              ├── sort_order: 排序（编号动态生成）
              ├── name: 证据名称
              ├── evidence_type: 书证/物证/电子数据...
              ├── direction: 我方/对方/法院调取
              ├── proof_purpose: 证明目的
              ├── three_properties: 三性说明 (JSON)
              ├── cross_examination: 质证意见 (JSON)
              ├── has_original: 是否有原件
              ├── original_location: 原件存放位置
              ├── page_count: 总页数
              ├── ocr_full_text: OCR 全文（用于搜索）
              └── EvidenceFile (证据文件)
                    ├── file: 文件
                    ├── file_hash: SHA-256
                    ├── source_channel: 来源渠道
                    ├── received_at: 接收时间
                    ├── page_count: 页数
                    └── ocr_text: OCR 文本

EvidenceGroup (证据组/争议焦点)
  └── M2M → EvidenceItem

EvidenceRelation (证据关联)
  ├── from_evidence → EvidenceItem
  ├── to_evidence → EvidenceItem
  └── relation_type: 关联类型

HearingNote (庭审笔记)
  ├── case → Case
  ├── content: 笔记内容
  ├── created_at: 记录时间
  └── M2M → EvidenceItem
```

### 3.2 功能优先级排序（P0-P3）

| 优先级 | 功能 | 价值 | 复杂度 |
|--------|------|------|--------|
| P0 | 证据上传与基本管理（CRUD） | 基础能力 | 低 |
| P0 | 证据清单自动生成（Word导出） | 每案必用 | 中 |
| P0 | 开庭模式（编号跳转+全文搜索） | 核心差异化 | 中 |
| P1 | AI 证据分类与证明目的建议 | 效率提升 | 中 |
| P1 | 自动页码计算 | 减少手工 | 低 |
| P1 | 多格式导出（合并PDF/分卷） | 电子提交必需 | 中 |
| P1 | 质证意见 AI 辅助生成 | 高专业价值 | 中 |
| P2 | 多渠道证据收件箱（邮件/飞书） | 自动化 | 高 |
| P2 | 证据分组与争议焦点关联 | 复杂案件 | 低 |
| P2 | 庭审笔记 | 实用辅助 | 低 |
| P3 | 证据关联图谱 | 可视化亮点 | 高 |
| P3 | 对方证据快速录入（拍照OCR） | 开庭应急 | 中 |
| P3 | 法院提交状态追踪 | 流程闭环 | 高 |

### 3.3 与现有模块的整合策略

| 现有模块 | 整合方式 |
|----------|----------|
| `evidence_sorting` | 作为证据分类的底层服务，被新的证据管理模块调用 |
| `documents` | 复用 docx 模板引擎生成证据清单 |
| `litigation_ai` | 复用 LLM 基础设施做证明目的建议和质证意见生成 |
| `cases` | 证据关联到案件，从案件当事人计算副本数量 |
| `automation` | 复用法院系统对接能力追踪提交状态 |
| `chat_records` | 复用飞书消息解析能力自动抓取证据图片 |

### 3.4 技术方案要点

1. **全文搜索**：SQLite FTS5，零额外依赖，微秒级查询
2. **文件哈希**：上传时计算 SHA-256，保证证据链完整性
3. **页码计算**：PDF 用 pikepdf 读页数，图片每张1页，累计计算
4. **PDF 合并导出**：pikepdf 合并 + reportlab 生成目录页
5. **开庭模式前端**：数据预加载 + 纯前端搜索，保证离线可用
6. **AI 功能**：复用 `litigation_ai` 的 LLM 调用链，新增证据相关 prompt

### 3.5 三个角色的最终共识

**律师**：系统最大的价值不是替代律师的专业判断，而是消除重复劳动（编页码、排序号、写格式化文本）和提供快速检索能力。开庭模式是杀手级功能——能在法官面前3秒内找到任何证据的律师，就是专业的律师。

**产品经理**：MVP 应该聚焦 P0 功能：上传 → 整理 → 导出 → 开庭检索。这四步形成完整闭环。AI 功能作为 P1 逐步叠加，每个 AI 功能都要有"人工确认"环节，不能全自动。

**程序员**：架构上新建 `evidence` app（不复用现有 `evidence_sorting`，后者保持为专用的对账单整理工具）。遵循四层架构，Model 层只定义字段，Service 层处理业务逻辑。全文搜索用 FTS5 虚拟表，通过 Django 的 raw SQL 访问。
